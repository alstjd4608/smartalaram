[스마트 알람 앱 - 통계 기능 SQLite 전환 가이드]

현재 알람 퀴즈 통계 데이터는 SharedPreferences와 JSON을 사용해 저장하고 있습니다. 
협업자분이 이 부분을 SQLite로 전환하실 때, 기존 화면단(UI) 코드 수정을 최소화하기 위한 협업 가이드입니다.

1. 목표
- 저장소 로직(QuizStatsStorage.java)을 SQLite 기반으로 변경
- 기존 UI (QuizActivity.java, StatsActivity.java) 코드는 최대한 수정 없이 그대로 사용

2. 가장 추천하는 방법: 기존 인터페이스 그대로 유지하기
협업자분이 SQLiteHelper를 연동하실 때, 기존 `QuizStatsStorage`의 메서드 구조를 똑같이 맞춰주시면 UI 코드를 전혀 수정할 필요가 없습니다.

[반드시 유지해야 할 메서드 2개]
public void addStat(QuizStatItem stat)
public List<QuizStatItem> loadStats()

[코드 예시]
public class QuizStatsStorage {
    private SQLiteHelper dbHelper; 
    
    public QuizStatsStorage(Context context) {
        dbHelper = new SQLiteHelper(context); // SQLite 초기화
    }
    
    public void addStat(QuizStatItem stat) {
        // DB에 INSERT 하는 로직 작성
    }
    
    public List<QuizStatItem> loadStats() {
        // DB에서 SELECT 해서 List<QuizStatItem> 형태로 리턴
    }
}

3. UI 코드를 직접 수정해야 하는 경우 (클래스 이름이 바뀐 경우)
만약 협업자분이 아예 새로운 이름(예: QuizSQLiteHelper)으로 클래스를 만드셨다면, 화면 단에서는 그 저장소를 호출하는 딱 두 곳만 이름을 변경해주시면 됩니다.

수정 포인트 1) QuizActivity.java (문제 풀고 데이터 저장하는 곳)
- 기존: statsStorage = new QuizStatsStorage(this);
- 변경: statsStorage = new QuizSQLiteHelper(this); // 새 클래스명으로 변경

수정 포인트 2) StatsActivity.java (통계 화면 보여주는 곳)
- 기존: QuizStatsStorage storage = new QuizStatsStorage(this);
- 변경: QuizSQLiteHelper storage = new QuizSQLiteHelper(this); // 새 클래스명으로 변경

결론:
입출력 형태인 `addStat()`과 `loadStats()` 구조만 똑같이 맞춰서 안의 내용물만 DB로 바꿔주시면 가장 빠르고 충돌 없이 병합(Merge)이 가능합니다!
